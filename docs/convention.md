# 開発規約（Conventions）

このリポジトリのブランチ・コミット・Issue の書き方を統一するための規約。
人間の共同作業者と Claude Code の双方がこれに従う。CLAUDE.md から `@docs/convention.md` で参照される。

## ブランチ名

```
<type>/#<issue番号>_<英語スネークケースの要約>
```

- `type` は以下から選ぶ： `feat` | `fix` | `refactor` | `docs` | `chore` | `test`
- Issue に紐づく場合は番号を入れる。無い場合は `#` ごと省略可（例 `chore/update_deps`）。
- 要約は英語・スネークケース・簡潔に。

例：
- `feat/#12_add_expiry_converter`
- `fix/#20_reject_invalid_month`
- `refactor/#8_extract_gs1_parser`

| type | 用途 |
|---|---|
| feat | 新機能 |
| fix | バグ修正 |
| refactor | 挙動を変えない内部改善 |
| docs | ドキュメントのみ |
| chore | ビルド・依存・設定など雑務 |
| test | テストの追加・修正 |

## コミットメッセージ

- **件名は日本語・簡潔（目安50字以内）**。誰が読んでも分かる平易な表現にする。
- 体言止め（「〜追加」）でも動詞（「〜を実装」）でもよい。表現を凝りすぎず、変更の要点を1行で。
- 本文は任意。**非自明な設計判断や「なぜ」を残したいときだけ**書く（何をしたかは diff で分かる）。
- Claude Code が関与したコミットは末尾に `Co-Authored-By` を付ける。

件名の例：
- `使用期限変換にバリデーション追加`
- `GS1パース機能追加`
- `未登録GTINのエラー表示を修正`

本文を書く例（設計判断を残す場合）：
```
使用期限変換にバリデーション追加

- 6桁数字/月(1-12)/日(月末超え)を検証し例外を投げる
- 日の上限は末日ベースで厳密判定（2/30・9/31 も弾く）
```

## Issue

```
タイトル: [type] 日本語で要約

## 背景
なぜ必要か・現状の課題

## やること
- [ ] タスク1
- [ ] タスク2

## 完了条件
どうなれば閉じてよいか
```

- タイトルの `type` はブランチと同じ語彙を使う（例 `[feat] 使用期限変換の実装`）。

## プルリクエスト

- **タイトル**は `[type] 日本語で要約`。`type` はブランチ・Issue と同じ語彙（feat/fix/refactor/docs/chore/test）。
- **粒度は「1 PR = 1 関心事」**。関心が分かれたら PR を分ける。
  - 例：ER図（docs）の追加と、その過程で見つけたスキーマ変更（schema.ts）は別 PR にする。混ぜるとレビューしづらく、片方だけ承認することもできない。
- **説明文は変更の大きさに比例させる**。小さな変更に長い説明は要らない。
  - diff を見た人が「なぜ？」と思う **非自明な点だけ** を言語化する。何をしたかは diff で分かるので書かない。
  - レビュー観点は、自分が迷っている所があるときだけ簡潔に添える（書きすぎない）。
- **Issue との紐付け**は本文に `関連: #<番号>` と書く。
  - `Closes #<番号>` / `Fixes #<番号>` は **その Issue を閉じてよいとき（Issue のタスクが全部終わるとき）だけ** 使う。サブタスクの PR では使わない（マージ時に Issue が自動クローズされるため）。

### テンプレートの使い方

type 別のテンプレを `.github/PULL_REQUEST_TEMPLATE/` に用意している。GitHub は PR では複数テンプレを自動で選ばせないので、作成時に明示指定する：

- CLI: `gh pr create --template docs.md`
- Web: PR 作成（比較）画面の URL に `?template=docs.md` を付ける

用意しているテンプレ: `feat.md` / `fix.md` / `docs.md`。他の type は必要になったら同じ形で足す。
